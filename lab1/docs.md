## Требование 10 — Поиск, фильтрация, сортировка, нечёткий поиск

### Что добавлено

#### 1. Файл `src/viewmodels/useResumeSearch.ts` (ViewModel)

Это custom hook (ViewModel в архитектуре MVVM), который инкапсулирует всю логику поиска, фильтрации и сортировки резюме. Хук принимает массив `Resume[]` из `useResumes` и возвращает обработанный (отфильтрованный, отсортированный) массив.

**Состояние хука:**
- `searchQuery` — текст поискового запроса
- `sortField` — поле сортировки (`'date'` или `'name'`)
- `sortOrder` — направление сортировки (`'asc'` или `'desc'`)
- `filterProfession` — выбранная профессия для фильтрации (или `null` = все)

**Алгоритм нечёткого поиска — расстояние Левенштейна (Levenshtein Distance):**

Расстояние Левенштейна — это метрика, определяющая минимальное количество односимвольных операций (вставка, удаление, замена), необходимых для превращения одной строки в другую. Например:
- `"kitten"` → `"sitting"` = 3 (замена k→s, замена e→i, вставка g)
- `"программист"` → `"програмист"` = 1 (удаление одной 'м')

Реализация использует классический алгоритм динамического программирования с матрицей размером `(m+1) × (n+1)`, где `m` и `n` — длины сравниваемых строк:

```typescript
function levenshtein(a: string, b: string): number {
    const m = a.length;
    const n = b.length;
    // dp[i][j] = расстояние между первыми i символами строки a
    //            и первыми j символами строки b
    const dp: number[][] = Array.from({ length: m + 1 }, () => Array(n + 1).fill(0));

    // Базовые случаи: превращение пустой строки в строку длины j
    for (let i = 0; i <= m; i++) dp[i][0] = i;
    for (let j = 0; j <= n; j++) dp[0][j] = j;

    for (let i = 1; i <= m; i++) {
        for (let j = 1; j <= n; j++) {
            const cost = a[i - 1] === b[j - 1] ? 0 : 1;
            dp[i][j] = Math.min(
                dp[i - 1][j] + 1,      // удаление
                dp[i][j - 1] + 1,      // вставка
                dp[i - 1][j - 1] + cost // замена (или совпадение)
            );
        }
    }
    return dp[m][n];
}
```

**Приоритеты поиска:**
1. Сначала проверяется **точное вхождение подстроки** (`includes`) — это быстро и точно
2. Если точного совпадения нет — используется **нечёткий поиск** (Levenshtein distance ≤ 2) по словам из текста

Поиск производится по четырём полям: `fullName`, `profession`, `email`, `skills`. Каждый терм (слово) из запроса должен найти совпадение хотя бы в одном поле.

**Фильтрация по профессии:**
Функция `getUniqueProfessions()` извлекает уникальные профессии из текущего списка резюме и возвращает их отсортированный массив. Пользователь может выбрать одну профессию для фильтрации.

**Сортировка:**
Сортировка реализована через `Array.sort()` с компаратором:
- По дате — сравниваются `Date` объекты из поля `createdAt`
- По имени — используется `String.localeCompare()` для поддержки кириллицы
- Направление (asc/desc) инвертирует результат компаратора

**Мемоизация через `useMemo`:**
Все вычисления (фильтрация + поиск + сортировка) обёрнуты в `useMemo`, который пересчитывает результат только при изменении зависимостей (`resumes`, `searchQuery`, `sortField`, `sortOrder`, `filterProfession`). Это предотвращает лишние пересчёты при ререндерах.

### Как протестировать и продемонстрировать

1. **Поиск по точному совпадению:**
   - Создать 3-5 резюме с разными именами и профессиями
   - В строке поиска ввести точное имя (например, "Иван") → должны отобразиться только резюме с именем "Иван"
   - Ввести профессию → должны отфильтроваться соответствующие

2. **Нечёткий поиск:**
   - Создать резюме с именем "Программист"
   - В поиске ввести "Програмист" (с опечаткой — одну 'м' пропустить) → резюме всё равно должно найтись (расстояние = 1 ≤ 2)
   - Ввести "Прогармист" (перестановка букв) → тоже должно найтись (расстояние = 2)
   - Ввести "Дизайнер" → не должно найти резюме "Программист" (расстояние > 2)

3. **Сортировка:**
   - Создать несколько резюме с разными именами и в разное время
   - Нажать кнопку сортировки "По дате" → резюме сортируются по дате создания
   - Нажать кнопку "По имени" → сортировка по алфавиту
   - Нажать кнопку направления → порядок инвертируется (A→Z ↔ Z→A)

4. **Фильтрация по профессии:**
   - Создать резюме с разными профессиями (например, "Разработчик", "Дизайнер", "Менеджер")
   - Нажать кнопку "Фильтр" → появится выпадающий список с профессиями
   - Выбрать "Разработчик" → остаются только резюме с этой профессией
   - Выбрать "Все профессии" → фильтр снимается

## Требование 11 — Настройка уведомлений по расписанию

### Что добавлено

#### 1. Файл `src/viewmodels/useNotifications.ts` (ViewModel)

Custom hook для управления уведомлениями.

**Используемые инструменты:**

- **`expo-notifications`** — библиотека из экосистемы Expo для работы с push-уведомлениями и локальными уведомлениями. Принцип работы:
  - Запрашивает разрешение у пользователя через системный диалог Android/iOS
  - Позволяет планировать уведомления через `scheduleNotificationAsync` с триггерами (по времени, интервалу, календарному расписанию)
  - На Android создаёт Notification Channel (обязательно с Android 8+) через `setNotificationChannelAsync`
  - Уведомления сохраняются в системном планировщике ОС и срабатывают даже при закрытом приложении

- **`expo-device`** — определяет, запущено ли приложение на физическом устройстве (`isDevice`). Уведомления не работают в эмуляторе Expo Go, поэтому перед планированием проверяется `Device.isDevice`

- **`AsyncStorage`** — key-value хранилище для сохранения настроек уведомлений:
  - `@notifications_enabled` — включены/выключены
  - `@notifications_hour` — час напоминания
  - `@notifications_minute` — минута напоминания

**Принцип работы:**

1. При включении тогглера вызывается `requestPermissionsAsync()` — показывается системный диалог запроса разрешения
2. При получении разрешения вызывается `scheduleNotificationAsync` с триггером `type: 'daily'` и указанным временем — ОС ставит ежедневное повторяющееся уведомление
3. При изменении времени предыдущие уведомления отменяются (`cancelAllScheduledNotificationsAsync`) и создаётся новое с обновлённым временем
4. При выключении — все запланированные уведомления отменяются

**Выбор времени:**
Реализован через степперы (кнопки ▲/▼): час меняется на ±1 (с переносом 23→0 и 0→23), минуты — на ±15 (0, 15, 30, 45).

#### 2. Изменения в `src/screens/SettingsScreen.tsx` (View)

Добавлена секция "Уведомления" (`settings.notifications`):
- **Switch** — включение/выключение ежедневного напоминания
- **Time Picker** — появляется при включённых уведомлениях, два блока степперов (часы:минуты) с кнопками вверх/вниз

#### 3. Изменения в `app.json`

Добавлен плагин `expo-notifications` в массив `plugins`. Это необходимо для Expo prebuild — плагин автоматически настраивает нативные модули (AndroidManifest.xml, Info.plist) при сборке.

#### 4. Изменения в `declarations.d.ts`

Добавлены TypeScript-декларации для модулей `expo-notifications` и `expo-device`, чтобы TypeScript знал о типах экспортируемых функций.

#### 5. Изменения в i18n файлах

Добавлены ключи:
- `settings.notifications` — заголовок секции
- `settings.notificationsEnabled` — метка переключателя
- `settings.notificationTime` — метка выбора времени
- `settings.notificationBody` — текст уведомления

### Как протестировать и продемонстрировать

1. **Включение уведомлений:**
   - Перейти в Settings → секция "Уведомления"
   - Включить тогглер "Ежедневное напоминание" → должен появиться системный диалог запроса разрешения (на физическом устройстве)
   - После подтверждения — появляется выбор времени

2. **Выбор времени:**
   - Нажимать стрелки ▲/▼ для часов и минут
   - Проверить, что часы переходят через 23→0 и 0→23
   - Минуты меняются шагом 15: 00→15→30→45→00

3. **Проверка уведомления:**
   - Установить время на 1-2 минуты вперёд от текущего
   - Свернуть приложение
   - Дождаться уведомления — должно прийти push-уведомление "Resume Builder: 📝"

4. **Отключение уведомлений:**
   - Выключить тогглер → уведомления отменяются
   - Time Picker исчезает
   - Ожидание в установленное время не должно приводить к уведомлению

5. **Сохранение настроек:**
   - Включить уведомления, установить время
   - Полностью закрыть и заново открыть приложение
   - Перейти в Settings → тогглер должен быть включён, время — тем же, что было установлено

## Требование 12 — Удалённая база данных (PostgreSQL + ImageKit)

### Что добавлено

#### 1. Файл `src/services/remoteApi.ts` (Сервисный слой)

REST API клиент для работы с удалённым PostgreSQL бэкендом.

**Используемые инструменты:**

- **`fetch` API** — встроенный в React Native HTTP-клиент. Принцип работы: отправляет HTTP-запросы к серверу и возвращает Promise с ответом. Поддерживает все методы (GET, POST, PUT, DELETE), заголовки и тело запроса

- **PostgreSQL** (на стороне сервера) — реляционная СУБД, которая хранит резюме в таблице. Бэкенд-сервер (Node.js/Express, Django, и т.д.) предоставляет REST API для CRUD-операций над этой таблицей

**Функции:**
- `fetchRemoteResumes()` — GET `/api/resumes` — получает все резюме с сервера
- `uploadResume(resume)` — POST `/api/resumes` — отправляет одно резюме
- `uploadResumes(resumes)` — POST `/api/resumes/batch` — пакетная загрузка всех резюме
- `deleteRemoteResume(id)` — DELETE `/api/resumes/:id` — удаляет резюме по ID

Базовый URL настраивается в константе `REMOTE_API_URL`.

#### 2. Файл `src/services/imageKitService.ts` (Сервисный слой)

Сервис для загрузки изображений в ImageKit.

**Используемые инструменты:**

- **ImageKit** — облачный сервис для хранения, оптимизации и доставки изображений. Принцип работы:
  1. Клиент запрашивает authentication-параметры (token, signature, expire) у бэкенда (endpoint `/api/imagekit/auth`). Это нужно, потому что приватный ключ ImageKit нельзя хранить на клиенте
  2. Клиент отправляет `multipart/form-data` запрос на `https://upload.imagekit.io/api/v1/files/upload` с файлом и auth-параметрами
  3. ImageKit сохраняет файл и возвращает URL для доступа
  4. URL можно использовать с трансформациями (ресайз, кроп, формат) через параметры запроса

- **FormData** — объект для формирования multipart-запросов. В React Native поддерживает `uri` файла, `type` (MIME) и `name`

**Функции:**
- `getAuthParams()` — получает подпись для загрузки с бэкенда
- `uploadImage(uri, fileName)` — загружает изображение из локального URI в ImageKit, возвращает URL
- `getImageUrl(path, transformations)` — формирует URL для доступа к изображению с опциональными трансформациями

#### 3. Файл `src/viewmodels/useRemoteSync.ts` (ViewModel)

Custom hook для синхронизации локальных данных с удалённым сервером.

**Состояние:**
- `syncing` — индикатор выполнения операции
- `lastSyncTime` — дата и время последней успешной синхронизации (сохраняется в AsyncStorage)
- `error` — текст последней ошибки (или `null`)

**Принцип работы:**

1. **Upload (syncToRemote):**
   - Проверяет наличие интернет-соединения через `useNetwork()`
   - Получает все локальные резюме из SQLite через `getResumes()`
   - Отправляет их пакетом на сервер через `uploadResumes()`
   - При успехе сохраняет время синхронизации

2. **Download (syncFromRemote):**
   - Проверяет интернет
   - Загружает резюме с сервера через `fetchRemoteResumes()`
   - Добавляет каждое резюме в локальную SQLite через `addResume()`
   - Сохраняет время синхронизации

Оба метода обрабатывают ошибки и показывают Alert с результатом.

#### 4. Изменения в `src/screens/SettingsScreen.tsx` (View)

Добавлена секция "Синхронизация" (`settings.sync`):
- **Кнопка "Выгрузить на сервер"** — отправляет локальные резюме на PostgreSQL бэкенд. Показывает ActivityIndicator при загрузке и время последней синхронизации
- **Кнопка "Загрузить с сервера"** — получает резюме с бэкенда и сохраняет в локальную БД
- **Блок ошибки** — при ошибке синхронизации отображается предупреждение с текстом ошибки

#### 5. Изменения в i18n файлах

Добавлены ключи:
- `settings.sync` — заголовок секции
- `settings.syncUpload` — метка кнопки выгрузки
- `settings.syncDownload` — метка кнопки загрузки
- `settings.lastSync` — метка времени последней синхронизации

### Как протестировать и продемонстрировать

#### Тест без бэкенда (демонстрация UI и обработки ошибок):

1. **Попытка синхронизации без сервера:**
   - Перейти в Settings → секция "Синхронизация"
   - Нажать "Выгрузить на сервер" → должен появиться ActivityIndicator (спиннер), затем ошибка подключения
   - Текст ошибки отображается красным в блоке ошибки

2. **Оффлайн-режим:**
   - Отключить интернет на устройстве
   - Нажать "Выгрузить на сервер" → должен появиться Alert "Offline — Cannot sync without internet connection"

#### Тест с бэкендом (полная демонстрация):

Для полного теста нужен REST API сервер. Пример минимального сервера (Node.js + Express + PostgreSQL):

```javascript
// Минимальный бэкенд для тестирования
const express = require('express');
const { Pool } = require('pg');
const app = express();
app.use(express.json());

const pool = new Pool({ connectionString: 'postgresql://user:pass@localhost/resumes_db' });

// Получить все резюме
app.get('/api/resumes', async (req, res) => {
    const { rows } = await pool.query('SELECT * FROM resumes ORDER BY id DESC');
    res.json(rows);
});

// Пакетная загрузка
app.post('/api/resumes/batch', async (req, res) => {
    const { resumes } = req.body;
    for (const r of resumes) {
        await pool.query(
            'INSERT INTO resumes (full_name, profession, email, phone, experience, education, skills) VALUES ($1,$2,$3,$4,$5,$6,$7)',
            [r.fullName, r.profession, r.email, r.phone, r.experience, r.education, r.skills]
        );
    }
    res.json({ ok: true });
});

app.listen(3000);
```

После запуска бэкенда:
1. Изменить `REMOTE_API_URL` в `remoteApi.ts` на адрес сервера (например, `http://192.168.1.100:3000/api`)
2. Создать несколько резюме в приложении
3. Нажать "Выгрузить на сервер" → должен появиться Alert "Resumes uploaded to server successfully"
4. Проверить таблицу в PostgreSQL — резюме должны быть там
5. Нажать "Загрузить с сервера" → Alert "Downloaded N resumes from server"
6. Время последней синхронизации обновляется под кнопкой

## Архитектура (MVVM)

Все изменения следуют архитектуре MVVM (Model-View-ViewModel):

```
┌─────────────────────────────────────────────────┐
│                    VIEW (Screens)                │
│  HomeScreen.tsx, SettingsScreen.tsx              │
│  — отображение UI, обработка жестов             │
│  — НЕ содержит бизнес-логику                    │
└────────────────────┬────────────────────────────┘
                     │ использует хуки
┌────────────────────▼────────────────────────────┐
│               VIEWMODEL (Custom Hooks)           │
│  useResumeSearch.ts — поиск/фильтрация          │
│  useNotifications.ts — уведомления              │
│  useRemoteSync.ts — синхронизация               │
│  — бизнес-логика и состояние                    │
│  — связь между View и Model                      │
└────────────────────┬────────────────────────────┘
                     │ вызывает
┌────────────────────▼────────────────────────────┐
│              MODEL (Services + DB)               │
│  db.ts — SQLite (локальная БД)                  │
│  remoteApi.ts — REST API (PostgreSQL)           │
│  imageKitService.ts — загрузка изображений      │
│  api.ts — внешний API (dummyjson)               │
│  — доступ к данным                              │
└─────────────────────────────────────────────────┘
```

**Принцип разделения:**
- **View** (экраны) знает только о ViewModel (хуках) и вызывает их функции
- **ViewModel** (хуки) содержит бизнес-логику и обращается к Model (сервисы, БД)
- **Model** (сервисы, БД) не знает о View и ViewModel — предоставляет чистые функции для работы с данными
